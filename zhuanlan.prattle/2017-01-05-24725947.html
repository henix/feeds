形而上最后还是要落地，作为一个程序员，首当其冲的就是代码命名问题。这是个终极问题，弄懂的同学可以接下来研究：

我是谁？我生从何来，死往何处，我为何要出现在这个世界上，我的出现对这个世界来说意味着什么，是世界选择了我，还是我选择了世界。我和宇宙有必然的联系吗？宇宙是否有尽头，时间是否有长短，过去的时间在哪里消失了，未来的时间又在何处停止，我在这一刻提出的问题，还是你刚才听到的问题吗？

言归正传。

我们都说名如其人，不要小看命名，糟糕的命名会导致糟糕的代码。我公众号第一篇阅读过万（汗）的文章：代码命名：僧敲月下门 讲的就是这个问题。我想，很多人读后有种於我心有戚戚焉的感觉，才力捧以及在朋友圈力荐它。

除了命名，另外一件但凡程序员都或多或少会遇到的事情就是重构。我特地为你准备了：代码重构之道，重构：撰写合格的代码。希望能够对你有所启发。

随着编程功力的提升，程序员需要随之提升的是 抽象的能力，以及 元编程的能力（来来来，咱们元编程入个门）。可惜，写到这里已经曲高和寡了。

（二）

如果要硬生生把程序员分成两个群体：会写代码的，不会写代码的。也可以这么分：调 API 的，写 API 的 —— 做 UI 的调完 SDK，调后端给的 API；做系统的一层层封装 API，最后包个 REST 的皮肤扔出去。其实广义说来，每个程序员即是写 API 的，也是调 API 的。每一个你写就的函数，都是一个 API。

撰写 API，尤其是 REST API，如今越来越成为程序员的一大主要工作。为此，我准备了一套全家桶，详细阐述我做 API 的心得：

  • 撰写合格的REST API （2015年旧文）

  • 再谈 API 的撰写 - 总览

  • 再谈 API 的撰写 - 架构

  • 再谈 API 的撰写 - 架构再谈 API 的撰写 - 子系统

  • 再谈 API 的撰写 - 契约

拿去不谢。2016 API 界比较重大的事件是 GraphQL 终于 production ready 了：GraphQL，你准备好了么？那么，是不是意味着 REST API 未来有可能被淘汰呢？我觉得几年内不会。未来更会发生的事情是 REST / GraphQL 并存。GraphQL 作为一层对外的窗口，包裹住 REST API。一个 GraphQL API，其数据源可能来自若干 REST API。二者互补。

ZeroMQ及其模式ZeroMQ 看上去是某个特定的应用，但其构建消息系统的思路和使用场景，如：pub / sub，router / dealer 等概念，一定要掌握。另外，都 2017 年了，如果还弄不明白 at least once / at most once 的消息传递机制，赶紧再翻回去看看。

程序员这一行，最怕的不是人老色衰，是未老先衰。技术从来都不是保值的，技术背后的知识才是驻颜术。当你要学习一门新的技术时，你该怎么做？希望 如何学习一门新的技术 能够帮到你。在过去的一年里，我遇到很多朋友后台留言询问学习某种技术的方法，尤其是 phoenix 在社区越来越火后，很多人寻思着放下手头学到一半的 rails，django，来学 phoenix。对此，我发了篇文章 rails, django, phoenix，你们错了 来阐述为何 web framework 不该是你学习甚至开发产品的中心。借用「中学为体，西学为用」的概念，我觉得「思想为体，框架为用」，切不可本末倒置，迷失在框架的茫茫苦海中跳脱不开。
