遥遥的原创、译作和沉思录

某天和一位负责客服工单的同事聊起手上的工作，讨论工单用途扩展的可能性。我提议：工单不仅可以用来跟进处理顾客提出的问题，还可以应用于公司内部的流程。


比如，线上机器报警，自动创建一张工单；业务希望提供一个支持，手动创建一张工单，技术人员都可以跟进。


同事提问：工单是服务业务专用的概念，为什么可以延伸到一般的工作概念上？我后来一想，原来我毕业于一家客户至上的公司，也就是亚马逊。在亚马逊内部，任何同事都可能成为你提供的服务的消费者，也就是你的客户。


客户有需求，就通过工单（Ticket）来提出。不管是业务对业务，还是业务对技术，甚至是技术对技术，需求的支持和跟进，都是通过工单的生命周期管理来完成。


其实在我现在的公司，也存在很承载工单类似功能的系统。例如，产品需求管理平台、运维需求管理平台、报警闭环流程等。


但是据我所见，也据我所了解，国内还没有像亚马逊一样把工单文化执行得如此统一和彻底的企业。我大概想出了两个的原因。


文化


亚马逊的领导力准则的第一条，就是顾客至上。据说 CEO 姐夫召开董事会，必定有一张椅子空着，代表未到场但最重要的成员：顾客。


也许这是个段子，但是亚马逊二十年来确实一直在使用一套全公司统一的工单系统，把服务客户的意识落实到公司大大小小的各个流程里面。这里的客户，不仅指外部的，也指内部的。


简单说来，业务是产品的客户，产品是技术的客户，同种职业里也把对方当成客户互相支持。配合强大的 DevOps 流程与工具，工程师也能通过工单系统直接支持业务方。


流程


亚马逊的工单其实跟常见的工单区别不大，可以大概用下面一个状态机来描述：



特殊之处在于齐全的周边设施。在亚马逊的工作流程中，任何事情只要有出错的可能，就一定会有工单被创建出来，就必须指定工单的接收者，就一定要有工单处理的过程和结论。


在亚马逊，每个系统都会把工单系统强制绑定成 error handler，要求错误处理的透明、闭环和复盘。


工单地址，就像是名片一般的存在。每个人、每个团队，都绑定了一个或多个工单地址。有问题找任何人？按这个地址提工单给他们就好了。下面是常见的情形（实际情形的数量多出几个数量级）：


人类提出的：


  • 功能支持

  • 数据支持

  • 功能报障

  • 业务关键点沟通

  • 新需求

  • 质疑撕逼

  • ...


机器提出的：


  • 自己的系统嗝屁了

  • 上下游系统挂了

  • 磁盘满了CPU高了

  • 证书要更新了

  • 机器要升级了

  • 程序要升级了

  • ...


不要怀疑一个普通工程师也能处理这些事，亚马逊有强大的 DevOps 工具可以让这些自助操作很轻松的完成，后面我会写文章讲到。


每一个工单的每个状态变更，都会有邮件通知。对于特别重要的情况，还会有系统发短信和打电话过来。


工单的完结，也一定要从预先设定好的原因分类中选出，并写明白问题产生的 root cause 和 solution。


我原来所在的部门 DevOps 文化很浓厚，每个月每位同事都有直接服务支持客户的机会，为期一周，几乎所有的沟通都通过工单来进行（除了电话沟通，IM沟通不算多）。大家对工单都有一种又爱又恨的复杂情感。


最后提一个有意思的地方。工单的事件升级机制是和组织结构绑定的，如果工单没有被及时处理，会按照组织架构逐级上报，一直报到 CEO。


例如，对于一个2级工单，要求半小时内必须要有人介入，2小时内必须妥善处理并降级。如果工单是半夜发出来的，你没有听到短信或邮件，那么来电叫醒你的，可能是和你一样睡意朦胧的上级，也可能是大洋彼岸的米国老板。


如果把工单看作一个个的 event，那么亚马逊算得上是名副其实的 event driven 组织架构了。
