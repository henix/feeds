原文

过去的几个月里我写作的主题越发偏向技术性的内容，这使得一些公众号的老读者感到不是很舒服，原本闲适时随手翻看一眼的内容，变作需伏在案几，细细琢磨的东西，甚至不写上几行代码，运行斟酌一下，都搞不明白我在讲什么。每周发表文章那日，订阅和取消订阅数量便开始急剧波动，有时一篇文章在不少新读者那里讨到了好彩头，却惹恼了更多的老读者，一进一出，反倒掉了不少关注。我记得公众号关注数量上万的时候，后台给出的女性读者的比例大概接近30%，如果未知性别的读者有一半是女性，那这个比例应该超过30%，昨日我看了一下后台的数据，这比例已经降到了不到20%，可见波动之剧烈。

写公众号已经有一年多半，起初只是想写点什么，顺便给自己的连载『途客圈创业记』寻个好去处，便开创了这个公众号，所以文字杂而不精，涉及的内容五花八门，创业，管理，产品分析，技术，读书，鸡汤，无所不包，所幸读者错爱，知乎谬赞，上了几次日报后，关注度便扶摇直上。在我飘飘然自我感觉良好之际热心读者点了句「知者不博，博者不知」，顿时惶恐。

文章贵专精，我开始学习舍得，收缩自己的写作阵线。

虽然亲手熬制的鸡汤却能捕获大量的读者，读书/创业也是人民群众喜闻乐见的专题，但我决定慢慢从这些内容中抽身而去，一来聊这些专题的作者实在太多，我即便视角再独到（大言不惭），也很难脱颖而出；二来我期待通过写作，能够 learning by teaching，而这些专题我并非边学边教，所以对我自己的裨益并不很大。

舍弃了大部分内容方向，留下的就是管理，产品分析以及技术。泛泛地介绍一个产品，我做不过engadget，techrunch，36kr，所以对于产品分析这个方向，我将其和技术融合在一起，即便是介绍一个产品，我也会尝试着分析其背后的技术，以技术为主。至于管理，虽然如今我并非身处管理岗位，但技术管理依然是我修习的方向，我也有大量的知识和经验需要学习，所以这个主题我依然会写。

一舍一留之下，「程序人生」的主力文章的方向便放在了技术和管理之上，而且，以技术为主，管理为辅，这样读者的定位也更加明确：他们是和我一样的对技术和管理孜孜以求的程序员们。服务好了这样的读者，才对得起「程序人生」这个名称。至于绝对的订阅数量的多寡，我虽依旧关心，但并不刻意追求，所谓：宠辱不惊，闲看庭前花开花落；去留无意，漫随天外云卷云舒。

前一段借着公众号支持语音素材的东风，我尝试做了六期的「图穷匕见」的podcast，占用时间不少，效果一般（阅读量均远低于同期的文章），况且与我聚焦的方向有所出入，所以这个内容无限期暂停，将来我也许会换一种形式重新做文字以外的探索（也许是MOOC，呵呵）。

接下来在写作上我会更加聚焦，话题收得更拢一些，每隔一段时间会列个提纲发表出来，督促自己按照提纲内容撰写文章。提纲所列内容一般是开放的主题 —— 它们并非全是我已经深入了解的内容，很多都是我一知半解，亟待进阶的内容。其实读过我近期es6系列，reactiveX系列文章的读者应该能够感觉到，那些文字是我在一边学，一边练的过程中总结出来的。

正巧10月份参加 aws re:invent 2015 时结识了 InfoQ 的 Mikey，他建议我在 InfoQ 开设专栏，这和我的想法不谋而合，某种程度上还能促使我在内容上少一分随心所欲（随波逐流），多一份严谨和认真。于是，接下来的多篇文章，在撰写完成后，我会先发于 InfoQ，再转载到公众号与知乎专栏上。计划中的内容分两大部分：

技术

最近会先写一系列AWS相关的文章，提纲如下：

  • 深入了解IAM和访问控制：介绍IAM，并从IAM中感悟访问控制之道。
  • 客户端身份验证的安全解决方案：介绍cognito。这是个我并不熟悉的领域，需要花时间学习。
  • Infrastructure as a code: CloudFormation：互联网软件的代码和其运行环境是密不可分的，运行环境不该是一条条命令行，一次次鼠标点击构建出来的，而应该是受到版本控制的一行行代码描绘出来的：它们可以被代码创建，可以被代码销毁，可以被一次性创建出多个相同的运行环境，部署不同的版本。
  • 使用API gateway 和 lambda构建REST API：这个话题我虽然聊过，但还可以深入下去，以一个实际的application来探讨其中的优劣。
  • S3：不仅仅是存储：这是我正在撰写的一篇文章，估计下周会发。等不及的读者可以关注我尚未完成的 github repo: tyrchen/aws-team-dropbox。在这篇文章里，我会尝试用 S3 创建公司内部的文件服务器，保存员工私人/共享文件，并以类似dropbox的方式双向同步。
  • 使用Kinesis流式处理实时数据：以一个实际的application来介绍如何使用Kinesis处理实时数据。
  • Dynamodb深度体验：DynamoDB看似简单，用起来则相当麻烦。本文将探讨如何进行DynamoDB的数据库（表）设计，如何使用 hash key / range key 使得查询更方便快捷？如何缓冲对DynamoDB的突发读写，如何自动scale up / scale down。
  • 使用AWS CLI操作资源和自动化：AWS CLI是一个强大的工具，这篇文章会尝试让你放弃AWS mangement console这样的GUI，而转至AWS CLI进行绝大多数的资源配置和自动化工作。
  • 在AWS上进行Continuously Delivery：这是一个相当开放的命题，我也依然在探索之中。
  • 在AWS上运行团队的IT系统：这篇文章会深入探讨一个团队如何在AWS上搭建完整的IT系统，从网络规划，到AWS内部服务的搭建（directory service，WorkMail，WorkDocs，CodeCommit，CodeDeploy），再到各种第三方的IT服务的搭建（vpn，bug tracking，project management tool，wiki，CI/CD）等。构建好这些工具，一个互联网创业团队应该能够正常运作了。（这篇文章可能会篇幅过长而不得不拆分成几篇）
  • 日志，监控以及可视化：介绍AWS对日志，监控和可视化的支持工具，并且详述第三方工具如何同CloudTrail一同协作。

这些文章并无前后依赖的关系，每篇都自成一体，因而我文章发出的顺序和上述提纲的顺序会有所出入。建议读者若要跟随文章来学习AWS，应该至少注册了一个AWS账号，事先阅读过当期所介绍服务的简介，并在AWS management console中尝试使用过该服务。否则，阅读的效果不会太好。

管理

在硅谷的日子里，我时不时参加一些meetup，参加一些聚会，这些聚会有时会有工程师介绍他们团队的文化和处世之道；我也会经常看各种有关创业团队文化的文章/视频，来充实自己的管理思想。以下是一些我打算撰写的主题：

  • 持续改进（continuously improvement）：敏捷的核心思想之一是持续改进，本文探讨团队如何进行持续改进，以及可用于持续改进的工具（思想）。
  • 深入解析Retrospective：retrospective是持续改进的一个重要工具。本文讲述retrospective的主要流程，以及一个成功的retrospective所具备的要素。
  • 在公司内部推行开源项目开发模式：本文探讨为何松散的开源软件项目对开发者的吸引力有时候会大于公司内部的项目，以及如何在公司内部创建一种开源软件开发的模式，提高软件开发的效率和效能。
  • 构建experiment-friendly的公司文化：创新是需要不断摸索的，一个公司的文化越有利于员工的探索，就会越有利于创新。本文介绍experiment-friendly的公司文化的意义，以及如何实施这样的公司文化。
  • 技术领导者的职责：如题，不解释。

上述技术文章和管理文章会穿插进行，如若能按照提纲写满 80% 的命题，相信对我自己，对读者都是不错的学习素材。

写了许久，改了许久，抱歉发晚了。
