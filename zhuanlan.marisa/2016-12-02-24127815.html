原文

事情的发生往往就那么巧, 这头 LambdaConf 发表了个 Standardized ladder of functional programming (有点类似程序员练级指南那种), 那头 @题叶 老师写了篇 为什么说 JavaScript 不擅长函数式编程

这两篇东西某种程度上都试图去定义 FP 是什么或者说描述 FP 应该有什么特性. 很多时候用 FP 这个词是个很模糊的分类, 一般介绍/宣传/安利语言的时候就会出现"函数式", "混合范式"这种字眼, 大家听得多用这些词来做分类方便交流也没有什么问题.

问题出在尝试以范式的角度讨论具体的问题, 如果题叶老师那篇 "为什么说 JavaScript 不擅长函数式编程" 改成 "为什么说 JavaScript 不能像 Haskell 那样写代码" bla bla 的说 Javascript 缺哪些特性不适合做什么事, 因为有哪个特性做什么事情做得更对能降低问题的复杂度, 这样讨论下来就是往解决问题方向靠, 而不是讨论是什么是 FP, FP 应该有什么特性这种对解决问题没有贡献的事情. 拿里面React的例子: om 搞了 mori -> 用不变性数据性能好了 -> Why -> 因为 bla bla -> immutable.js, 这样关注点和讨论就在不变性数据上, 而不是哎呀 Haskell 默认都 immutable 很 FP, "immutable 是 FP 的精粹/思想" bla bla. 这里没有批评题叶老师的意思, 一直在关注题叶老师的写的东西和分享, 他有很多很有意思的想法我很佩服, 换种表达方式的话可能会让这些想法让更多人理解.

OOP 其实也有这种尿性, Java 的 OO, Ruby 的 OO, MOP 的 OO 不怎么能好讨论到一堆去, 后来看到有人说要针对某个语言的对象模型(不是在内存那种)来讨论, 例如说, "多重继承"不好啊会有钻石问题, Ruby 的 mixin 爽在多重继承上退一步, 但是又可以做到多重继承那种复用.


FP 某种程度上成为了新时代的 OOP, 不是说地位或者市场份额的替代, 而是buzzword generator 的替代, 尤其搭上前端这个潮流, 看下 Standardized ladder of functional programming 里各种词, 是不是有种 23 种 design pattern的感觉.
